# Сценарий защиты проекта “Аиша”

## Что сказать в начале

Я разработал Telegram-бота “Аиша” для помощи абитуриентам. Бот помогает подобрать вуз по региону, баллам ЕГЭ, направлению и типу обучения, показывает категории поступления, позволяет сохранить интересные варианты, сравнить их и открыть Mini App.

Проект демонстрационный, но основной пользовательский сценарий сейчас PostgreSQL-first: при `USE_POSTGRES=true` backend берёт каталог и справочники из SQL-базы команды, а JSON-база остаётся fallback для локального запуска без PostgreSQL. В обоих режимах бот и Mini App работают через тот же `/api/universities`. На текущем этапе backend также отдаёт справочники регионов, городов, направлений, форм обучения, типов конкурса и индивидуальных достижений.

Короткая формулировка для защиты: мы не просто подключили PostgreSQL, а проверили всю SQL-схему команды, зафиксировали, какие таблицы реально участвуют в пользовательском сценарии, и сделали справочники/поиск так, чтобы Mini App работал по полной базе, а не только по первым подсказкам интерфейса.

## Что сказать про данные

Основной сценарий защиты использует PostgreSQL-режим: при `USE_POSTGRES=true` backend берёт каталог вузов, направлений, факультетов, проходных баллов и индивидуальных достижений из SQL-базы команды по `DATABASE_URL`.

```bash
source .venv/bin/activate
export USE_POSTGRES=true
export DATABASE_URL="postgresql://$(whoami)@localhost:5432/tgbot"
export BACKEND_BASE_URL="http://localhost:8000"
bash scripts/run_backend.sh
```

JSON fallback `backend_stub/data/universities.json` остаётся резервным режимом для локального запуска и тестов без live-базы. В обоих режимах Telegram-бот и Mini App работают через тот же backend API, поэтому основной код поиска, сравнения, summary и Mini App менять не нужно.

На этапе 45 сделан отдельный документ `docs/DATABASE_USAGE.md`: в нём перечислены таблицы `universities`, `faculties`, `directions`, `passing_scores`, `achievements`, пользовательские таблицы исходной схемы и view `v_directions_with_latest_budget`. Это показывает, что команда не использует базу вслепую: каждое поле либо участвует в API/UI, либо имеет понятную причину, почему оно пока не подключено.

Для PostgreSQL-режима отдельно проверена совместимость данных: старые поля API сохраняются, новые `year`, `faculty`, `admission_type`, `financing_label`, `contest_label`, `study_form_label`, `note`, `profile` используются как полезные детали, а диагностическое `source` не показывается пользователю. Если в SQL нет предметов, стоимости или срока обучения, интерфейс скрывает эти строки, а не выводит `None`, `null`, `undefined` или `[]`.

При проверке PostgreSQL-данных были найдены technical/synthetic записи из supplemental seed, например `Региональный центр технологий и инженерии (...)` с кодами `РЦТИ-12`. Это реальные строки `universities.name`, а не ошибка JOIN. Они не удаляются из базы, но фильтруются в пользовательской выдаче по умолчанию; для диагностики есть `include_synthetic=true`.

Интерфейс использует корректные подписи: `Финансирование` для бюджет/платное, `Форма обучения` для очной/заочной формы и `Конкурс` для квот. `Конкурс: бюджет` не выводится как дубль к финансированию.

Также добавлены алиасы под реальные данные команды: `Крым` ищется как `Республика Крым`, `Адыгея` как `Республика Адыгея`, старое направление `IT` раскрывается в реальные направления PostgreSQL, а ввод вроде `09.03.04 Программная инженерия` ищется по коду, названию и профилю. Поэтому старые демонстрационные сценарии не дают пустую выдачу только из-за другого словаря в SQL.

Для справочника направлений есть отдельный endpoint `GET /api/directions?q=...`: Mini App показывает ограниченный стартовый список подсказок, но поиск по коду `09.03.04`, `архитектура` или `экономика` идёт по полной PostgreSQL-базе.

Справочник направлений большой, поэтому в Mini App сделан поиск по полному PostgreSQL-справочнику с удобными подсказками и кнопкой очистки выбранного направления. Пользователь может выбрать направление, нажать `×`, сразу ввести новый код или название и выбрать другой вариант без очистки всей формы.

Короткие технические названия вроде `РЦТИ-26` не выводятся как главный заголовок карточки: UI берёт полное `universities.name`, а короткое название показывает отдельно только если оно похоже на нормальную аббревиатуру.

На реальных данных PostgreSQL встречаются специальные значения проходных баллов для квот или неполных записей. Интерфейс не показывает `min_score < 40` как обычный проходной балл и не считает запас от таких значений; вместо этого показывает, что балл требует уточнения, и выводит контекст конкурса, года и примечания.

Справочник `achievements` в SQL не привязан к вузам, поэтому `/api/achievements` показывает его как общий список индивидуальных достижений. Это честнее, чем приписывать такие баллы конкретным университетам без связи в схеме.

Логика баллов сохраняет амбициозный зазор прототипа: в выдачу попадают варианты с `min_score <= score + 20`, чтобы пользователь видел программы немного выше текущего результата.

## Что сказать про OpenAI

OpenAI опционален.

Если `OPENAI_API_KEY` пустой, бот работает через локальный fallback:

- подбор вузов работает;
- сравнение работает;
- summary работает;
- advice работает локально и не требует OpenAI;
- filters работают локально с последними результатами;
- export работает локально по последнему подбору и избранному;
- history работает и хранит только краткое описание последних подборов;
- психологическая поддержка работает готовыми безопасными ответами.

Safety/crisis-фразы не отправляются в OpenAI.

## Что сказать про меню Telegram-бота

Когда функций стало много, главное меню было переработано. На первом экране остались только основные разделы: `🎓 Подобрать вуз`, `📱 Mini App`, `📌 Мои результаты`, `🤝 Помощник`, `⚙️ Сервис`, `ℹ️ О проекте`.

После этого меню было визуально оформлено как карточки: бот отправляет локальный баннер, короткое описание и inline-кнопки. Это делает демонстрацию аккуратнее и не перегружает пользователя списком всех функций сразу.

Действия с последним подбором разделены по назначению. В `📌 Мои результаты` находятся избранные вузы, сравнение, экспорт, история и итог. В `⚙️ Сервис` находятся фильтры, объяснение категорий, приватность, сброс данных, BotFather и обратная связь. В `🤝 Помощник` вынесены мягкие support-сценарии и короткий план.

После поисковой выдачи бот больше не показывает весь список функций сразу. Там остаются только действия, которые нужны в моменте: максимум 5 кнопок `⭐ Сохранить N`, `➡️ Ещё варианты`, `📌 Мои результаты`, `🎓 Новый подбор`, `🔙 Главное меню`. Если вариантов больше 5, они показываются страницами: `1–5`, `6–10`, `11–15` и так далее.

В Telegram-боте регион и направление вводятся текстом, потому что база содержит много регионов и направлений. Кнопки используются только там, где выбор фиксированный: финансирование и действия после выдачи результатов.

Все команды при этом сохранились: пользователь может открыть функции как через меню, так и напрямую через `/summary`, `/favorites`, `/filters`, `/export`, `/feedback`, `/support`, `/plan` и другие команды.

Если Telegram не принимает баннер или файл недоступен, сервис `menu_cards.py` отправляет текстовый fallback с теми же кнопками. Поэтому навигация не зависит от успешной отправки изображения.

## Что сказать про Mini App

Mini App находится в папке `mini_app/` и сделан без frontend-фреймворков: HTML + CSS + JS.

Он работает через тот же backend endpoint:

```text
/api/universities
```

Локально Mini App открывается через:

```text
http://localhost:8000/miniapp
```

Для Telegram нужен публичный HTTPS URL через `WEBAPP_URL`.

В Mini App есть вкладки, локальные фильтры результатов, компактный итог подбора, визуальная таблица сравнения, избранное и переключение светлой/тёмной темы. Если Mini App открыт через Telegram, backend проверяет WebApp `initData`, Mini App показывает статус проверенной сессии, а избранное синхронизируется с Telegram-ботом. Сравнение и тема остаются локальными.

Справочники формы Mini App загружаются из backend: регионы, города, направления, формы обучения, типы конкурса и индивидуальные достижения. Если backend недоступен, Mini App включает fallback-справочники и показывает это в статусе, чтобы пользователь понимал источник данных.

Поле направления не использует системный datalist как единственный способ выбора: это кастомный picker с dropdown, поиском `/api/directions?q=...&limit=20`, пустым состоянием и клавиатурной навигацией. Это важно, потому что PostgreSQL-справочник направлений слишком большой для обычного длинного списка.

Для быстрой демонстрации в Mini App есть блок `Быстрый старт`: можно нажать готовый сценарий, например `Адыгея · 230 · IT · Бюджет`, и приложение само заполнит форму, запустит поиск и покажет toast-уведомление. Также в форме есть подсказки, очистка формы, понятные пустые состояния и toast-обратная связь при сохранении, сравнении и переключении темы.

Mini App получил фирменный animated-brand в верхней панели: `Aisha` появляется побуквенно через CSS-анимацию без внешних шрифтов и библиотек, но не дублирует hero-заголовок. Рядом с основным интерфейсом есть блок `Aisha советует`: он даёт короткие контекстные подсказки по текущему состоянию подбора, избранного, сравнения, плана, экспорта или поддержки. Подсказки можно скрыть, поэтому интерфейс остаётся ненавязчивым.

Mini App не просто показывает список вузов, а формирует персональный план действий: сохранить варианты, сравнить их, проверить официальные сайты и выбрать безопасные, реалистичные и амбициозные направления. План строится локально по последнему подбору, избранному и сравнению, поэтому backend API не меняется.

Mini App также позволяет подготовить отчёт по подбору: параметры, найденные варианты, избранные вузы, сравнение и план поступления. Отчёт можно скопировать как текст или сохранить как PDF через системное окно печати браузера. PDF не генерируется программно, поэтому новые зависимости не нужны.

На защите можно показать, что Mini App переключается между светлой и тёмной темой без перезагрузки страницы и запоминает выбор после обновления.

Также можно показать визуальное сравнение: выбрать 2–3 вуза из результатов или избранного и открыть вкладку `Сравнение`, где Mini App сравнит программы по баллам, категории, типу обучения, стоимости, форме, сроку и сайту без повторного запроса к backend.

Можно показать синхронизацию избранного: сохранить вуз в Telegram-боте, открыть Mini App через `/webapp` и увидеть этот вуз во вкладке `Избранное`. Затем добавить вуз в Mini App и проверить, что он появился в Telegram-боте. Важно объяснить, что user_id не передаётся из клиента напрямую, а достаётся backend из проверенного Telegram WebApp `initData`.

Можно отдельно показать диагностику сессии: открыть Mini App в обычном браузере и увидеть режим запуска `Браузер`, затем открыть через Telegram `/webapp` и увидеть, что сессия проверена. Источник данных показывается отдельно, поэтому браузерный запуск не означает JSON fallback. Это демонстрирует, что локальная разработка не ломается, но синхронизация включается только для проверенной Telegram-сессии.

Можно показать обратную связь: пользователь отправляет обращение через Telegram-бота или вкладку `Поддержка` в Mini App, получает номер заявки `AISH-0001`, а `/my_feedback` показывает последние обращения. Это демонстрационная локальная система: обращения сохраняются в `telegram_bot/storage/feedback.json`, без email, админ-панели и внешней базы.

### Почему проверка Mini App-сессии безопаснее

- `user_id` не принимается из query или body;
- Telegram WebApp `initData` проверяется на backend;
- HMAC считается через bot token, который не отправляется в Mini App;
- `hash` сравнивается через `compare_digest`;
- старые `auth_date` можно отклонять;
- без валидной сессии Mini App работает только локально и не получает доступ к избранному пользователя в боте.
- user_id для обращений Mini App берётся только из проверенного `initData`, а не из query/body клиента.

## Возможные вопросы преподавателя

### Что делает проект?

“Аиша” помогает абитуриенту быстро собрать список программ вузов по региону, баллам, направлению и финансированию. Пользователь видит категории вариантов, может сохранить интересные программы, сравнить их, получить краткий план действий и открыть тот же подбор в Mini App.

### Почему Telegram-бот?

Telegram удобен для пошагового сценария: бот задаёт короткие вопросы, хранит последний подбор и избранное, даёт кнопки сохранения и не требует установки отдельного приложения. Это снижает порог входа для абитуриента.

### Чем Mini App отличается от обычного бота?

Бот удобен для диалога, а Mini App удобен для визуальной работы: вкладки, фильтры, сравнение, избранное, план и экспорт отчёта видны на одном экране. Mini App использует тот же backend API, поэтому подбор в чате и в веб-интерфейсе согласован.

### Где хранится база вузов?

При `USE_POSTGRES=true` каталог берётся из PostgreSQL по `DATABASE_URL`. Без PostgreSQL используется JSON fallback `backend_stub/data/universities.json` для локального запуска и тестов.

### Почему оставлен JSON fallback?

Fallback нужен, чтобы проект запускался на ноутбуке без live PostgreSQL и чтобы тесты не зависели от локальной базы. Это резервный режим, а не основной пользовательский сценарий защиты.

### Что будет, если OpenAI ключа нет?

Бот продолжит работать через fallback. OpenAI не обязателен.

### Что делает backend?

Backend принимает параметры поиска, нормализует ввод и возвращает JSON-список программ. Источник данных выбирается настройкой: JSON fallback по умолчанию или PostgreSQL при `USE_POSTGRES=true`. Фильтрация по региону, городу, направлению, типу, форме обучения, году, общему поиску `q` и сортировка выполняются на backend.

### Зачем Mini App, если есть Telegram-бот?

Mini App показывает тот же подбор в визуальном веб-интерфейсе. Это удобно для демонстрации и дальнейшего развития проекта.

### Зачем PostgreSQL?

PostgreSQL хранит каталог вузов в нормализованных таблицах: вузы, направления, факультеты и проходные баллы. Благодаря этому backend может строить справочники из базы, фильтровать по нескольким полям и сортировать выдачу без изменения Telegram-бота и Mini App.

### Вы используете всю базу команды?

Мы используем все таблицы, которые нужны текущему пользовательскому сценарию: `universities`, `faculties`, `directions`, `passing_scores` и `achievements`. Пользовательские таблицы исходной схемы не подключены намеренно, потому что Telegram-бот уже хранит профиль, избранное и feedback в своём JSON storage и проверяет Mini App-сессию через Telegram `initData`. Это зафиксировано в `docs/DATABASE_USAGE.md`.

### Как работает подбор?

Backend нормализует регион, направление, код направления и финансирование, затем ищет подходящие строки через связку `passing_scores -> directions -> universities` и `LEFT JOIN faculties`. Вариант попадает в выдачу, если он соответствует фильтрам и его валидный балл не выше пользовательского больше чем на 20 баллов.

### Какие таблицы участвуют в выдаче?

Для карточек используются `universities`, `directions`, `faculties` и `passing_scores`. Для справочника индивидуальных достижений используется `achievements`, но она не привязана к конкретным вузам в SQL-схеме, поэтому показывается как общий справочник.

### Почему достижения общие?

В таблице `achievements` нет связи с `university_id` или `direction_id`. Поэтому проект не приписывает эти баллы конкретным вузам и честно показывает их как общий список: точные правила нужно проверять на сайтах приёмных комиссий.

### Как работают категории?

Категории считаются по валидному проходному баллу: safe означает заметный запас, realistic - близкий балл, ambitious - небольшую нехватку в пределах 20 баллов. Это не гарантия поступления, а ориентир для сравнения вариантов.

### Почему нет гарантии поступления?

Проходные баллы и правила приёма меняются каждый год, а часть SQL-значений может относиться к квотам или минимальным порогам. Поэтому бот всегда оставляет предупреждение: финальные условия нужно сверять на официальных сайтах вузов.

### Почему регион и направление вводятся текстом?

PostgreSQL-справочники регионов и направлений большие. Текстовый ввод быстрее и не перегружает Telegram-клавиатуру, а backend понимает алиасы вроде `Крым`, `Адыгея`, `айти` и коды направлений `09.03.04`.

### Почему финансирование кнопками?

Финансирование имеет маленький фиксированный набор значений, поэтому на этом шаге удобнее reply-клавиатура: `Бюджет`, `Платное`, `Любое`, `Назад`. Выбор `Любое` не отправляет `type` в backend и не сужает выдачу.

### Как работает кнопка “Ещё варианты”?

Telegram-бот показывает результаты страницами по 5. На первой странице доступны `⭐ Сохранить 1` ... `⭐ Сохранить 5`, на второй - `⭐ Сохранить 6` ... `⭐ Сохранить 10`; `active_results` обновляется под текущую страницу, поэтому сохраняется именно показанный вариант.

### Как работает сохранение?

Сохранение берёт текущую показанную порцию результатов из `active_results` и добавляет выбранный вариант в локальное JSON-хранилище пользователя. Полный `last_results` при этом не теряется и используется для summary, compare, history и export.

### Как работает сравнение?

Бот берёт последние результаты или избранные вузы и сравнивает минимальный балл, категорию, финансирование, стоимость, предметы, форму обучения и срок.

### Как работает история подборов?

После каждого поиска бот сохраняет краткую запись: регион, баллы, направление, финансирование, количество результатов, категории и топ-3 варианта. Хранится максимум 5 последних записей в локальном JSON storage.

### Как работают советы по подбору?

Команда `/advice` берёт последний подбор, считает баланс безопасных, реалистичных и амбициозных вариантов и предлагает следующий шаг: сохранить вузы, сравнить варианты, проверить сайты, расширить регион, направление или финансирование. OpenAI для этой функции не используется.

### Как работают фильтры?

Команда `/filters` берёт последний подбор и показывает фильтры: все, безопасные, реалистичные, амбициозные, бюджетные и платные варианты. Для сохранения после фильтра используется отдельное поле `active_results`, поэтому полный `last_results` остаётся доступен для `/summary`, `/history` и `/compare`.

### Как работает экспорт?

Команда `/export` берёт последний профиль, `last_results` и избранные вузы, формирует plain text отчёт и может отправить его сообщением или `.txt` документом. Export-файлы не сохраняются постоянно в проекте.

Во вкладке `Экспорт` Mini App отчёт строится локально по состоянию веб-интерфейса: последний подбор, избранное, сравнение и персональный план. Его можно скопировать или сохранить как PDF через печать браузера.

### Что такое safe/realistic/ambitious?

- safe - есть заметный запас по баллам;
- realistic - баллы близки к минимальному уровню;
- ambitious - немного не хватает, но вариант можно рассматривать как цель.

Это не гарантия поступления.

### Как проверить PostgreSQL перед защитой?

```bash
source .venv/bin/activate
export USE_POSTGRES=true
export DATABASE_URL="postgresql://$(whoami)@localhost:5432/tgbot"
python3 scripts/check_postgres.py
```

Скрипт проверяет схему, покрытие данных, synthetic rows, suspicious scores, справочники, q-search направлений и API-like сценарии.

### Почему используется ngrok?

Telegram Mini App требует публичный HTTPS URL. Ngrok даёт временный HTTPS адрес для локальной демонстрации.

### Почему проект не работает 24/7 без сервера?

Сейчас backend, бот и ngrok запускаются локально на ноутбуке. Если ноутбук выключен или ngrok остановлен, Telegram не сможет открыть Mini App и бот не будет отвечать. Для постоянной работы нужен VPS или другой сервер.

### Что будет следующим шагом после защиты?

Следующий практичный шаг - перенести backend и bot runner на VPS, подключить постоянный HTTPS-домен для Mini App, настроить systemd/docker и регулярный backup пользовательских JSON-хранилищ или полноценную серверную таблицу пользователей.

### Как заменить fallback-данные или расширить источник?

Можно заменить `backend_stub/data/universities.json` финальным файлом с той же структурой полей или загрузить SQL-файлы в PostgreSQL и включить `USE_POSTGRES=true`. Для JSON fallback проверки такие:

```bash
python scripts/check_data.py
bash scripts/check_project.sh
```

Формат записей описан в `docs/DATA.md`. Если проверка проходит, логика бота и Mini App останется прежней, потому что они работают через единый backend API.
